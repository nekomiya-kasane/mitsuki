## Gap Analysis

### Why This Task Exists

Phase 2's demo deliverable (`forward_cubes` at 60fps on all 5 backends) and every subsequent
Phase's demo require rendering to a visible window. However:

- Roadmap L1052-1054 explicitly defer `CreateForWindow` (Vulkan/D3D12/WebGPU swapchain) to **Phase 12**.
- Phase 12's `RenderSurface` is a heavy multi-window abstraction with per-view render graphs.
- No intermediate "single-window present" was planned between Phase 1a and Phase 12.
- `GlfwBootstrap` currently uses `CreateOwned()` (headless) + `glfwSwapBuffers()` (no-op for `GLFW_NO_API`).
- Triangle demo interactive path renders to `OffscreenTarget` — window is always black.

This is a **minimum viable present** task — NOT the full Phase 12 `RenderSurface`.

### Architecture Decision: Per-Frame-In-Flight Synchronization

**Decision date**: 2026-03-12
**Industry research**: Diligent Engine, Google Filament, wgpu, bgfx, The Forge

Original implementation used a single pair of semaphores (`imageAvailableSemaphore_` + `renderFinishedSemaphore_`) with no fence-based CPU throttle. This is incorrect — all production frameworks use per-frame-in-flight arrays.

**Upgraded to Mode A (Diligent/Filament/wgpu pattern)**:

```
VulkanSwapchain {
    static constexpr kMaxFramesInFlight = 2;

    VkSemaphore  imageAvailableSemaphores_[2];   // per-frame GPU↔GPU: acquire → submit
    VkSemaphore  renderFinishedSemaphores_[2];   // per-frame GPU↔GPU: submit → present
    VkFence      inFlightFences_[2];             // per-frame CPU↔GPU: throttle
    uint32_t     currentFrame_ = 0;              // rotates in Present()
}
```

**Synchronization flow per frame**:
1. `AcquireNextImage()`: `vkWaitForFences(inFlightFences_[currentFrame_])` → `vkResetFences` → `vkAcquireNextImageKHR`
2. `GetSubmitSyncInfo()`: returns `{waitSem, signalSem, fence}` for current frame slot
3. `IDevice::Submit(cmdBuf, syncInfo)`: `vkQueueSubmit(..., fence)` with wait/signal semaphores
4. `Present()`: `vkQueuePresentKHR(..., renderFinishedSem)` → `currentFrame_ = (currentFrame_ + 1) % 2`

**API changes**:
- `SubmitSyncInfo` (in `RhiDescriptors.h`): added `uint64_t signalFence` field (opaque `VkFence` for Vulkan, ignored by other backends)
- `IDevice::Submit(cmdBuf, syncInfo)`: second parameter added with default `{}` — backward compatible, all existing call sites unchanged
- `VulkanDevice::Submit()`: passes `signalFence` to `vkQueueSubmit` fence parameter
- All 5 backend implementations updated (`VulkanDevice`, `D3D12Device`, `OpenGlDevice`, `WebGpuDevice`, `MockDevice`)

**Downstream impact**:
- T2.8.1 (forward_cubes demo): must call `swapchain->GetSubmitSyncInfo()` and pass to `device->Submit()`
- Phase 12 (RenderSurface): `SubmitSyncInfo` is forward-compatible infrastructure; `RenderSurface` can internally manage sync arrays and inject sync info into submit
- Existing tests: unaffected (default `SubmitSyncInfo{}` = no semaphores, no fence = same as Phase 1a behavior)
